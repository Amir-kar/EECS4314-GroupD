Abstract

This report provides a concrete architecture analysis and a deep dive into YARN (Yet Another Resource Negotiator), the resource management and job scheduling framework of Hadoop. Unlike the previous report which was centered around the conceptual study, this report is derived from the actual implementation of Apache Hadoop using SciTools Understand to identify dependencies and architectural patterns within Hadoop’s implementation.

        Our concrete study has confirmed the four main components of Hadoop: Common, HDFS, YARN, MapReduce, but also revealed that the actual dependencies between modules differ from those found in the documented conceptual model.  Two particular examples are, YARN depends on HDFS for Distributed file system configuration keys, and the Common depends on HDFS for a header file needed if YARN is running on Windows OS. Both of these dependencies were not assumed from our findings through Hadoop’s documentation and online resources.

        YARN’s concrete architecture demonstrates clear layering(Client, Server, Support) and puts Master-Slave, Client-Server, Distributed Fault Tolerance, MicroServices, and Event-Driven styles through  RPC. This report highlights how Hadoop’s concrete architecture supports modularity, scalability, and distributed fault tolerance, while also comparing it to the conceptual model to illustrate key deviations and rationales behind Hadoop’s as-built structure.

From Conceptual to Concrete: Hadoop Architecture Overview

        Apache Hadoop is composed of four main subsystems: Common, HDFS, YARN, and MapReduce. A layering is formed in a top down manner that is visually explained in the conceptual architecture shown below.

        

A quick recap on the roles of Hadoop’s main subsystems:

  • Common provides shared libraries
  • HDFS stores and replicates data blocks across nodes.
  • YARN manages cluster resources
  • MapReduce executes distributed computation.

Hadoop Conceptual Architecture

Conceptually, the interactions between these layers are defined as the following:

  • The Common <-> HDFS, YARN, MapReduce: Common provides libraries, services, utilities that HDFS, YARN and MapReduce import and use.
  • HDFS <-> YARN: YARN stores job-related data such as application JARS, logs and outputs in HDFS. YARN also retrieves application data and configurations from HDFS when launching jobs.
  • HDFS <-> MapReduce: MapReduce reads input from HDFS blocks and writes output back to HDFS. HDFS provides data locality information so MapReduce tasks can be executed near the data blocks.
  • YARN <-> MapReduce: YARN manages the containers for MapReduce tasks to run on.

        However, the concrete architecture we found in SciTools Understand shows us a more conjoined set of dependencies that deviate from the straight-forward conceptual layering we previously found.

Hadoop Concrete Architecture

Concretely, the interactions between these layers are defined as the following:

  • YARN -> HDFS: Other than for resources, YARN uses HDFSConfiguration, DFSConfigKeys, and Delegation tokens for runtime dependencies, logging, and security.
  • Common -> HDFS: Common uses a header file “unistd.h” provided by HDFS for compiling some of its libraries on Windows OS. This is because Windows OS is not POSIX compliant and does not provide “unistd.h” for Common like other POSIX OS
  • Testing dependencies: YARN makes references to MapReduce during testing.  And HDFS depends on YARN for an integration test in its test package: HDFS imports RM_PRINCIPAL from YarnConfiguration in TestRouterFederationRenameInKerberosEnv.java.

Our findings found in Hadoop’s concrete architecture indicate that the actual implementation includes coupling between subsystems which are used for testing, and specific configuration or OS dependent quirks.  These bidirectional dependencies indicate that in-practice, Hadoop prioritizes other factors over strict decoupled layering.

From Conceptual to Concrete: YARN Architecture Overview

YARN (Yet Another Resource Negotiator) is the component responsible for resource management and jobs scheduling in Hadoop. There are 4 key components in YARN: Resource Manager, Application Master, Node Manager and Container.

  • Resource Manager(RM): There is only 1 Resource Manager in YARN. It is the master daemon for allocating resources and scheduling jobs. The RM schedules and allocates containers.
  • Application Master(AM): There is 1 Application Master per application. It is responsible for managing the full life cycle of its application.The AM requests for containers, and orchestrates the NM to launch and stop
  • Node Manager(NM): There is 1 Node Manager per node and an application is typically within containers located in multiple nodes. The NM is responsible for managing its node and the containers within itself.
  • Container: There are multiple Containers in a node, each runs a task. A container is a resource bundle (CPU, Memory) where a task runs.

YARN Conceptual Architecture

        At the conceptual level, YARN’s operation is defined by the following dependencies:

Dependencies:

  • AM -> RM: Depends on RM for allocating an initial AM container for itself and for additional container allocations for the app’s tasks.
  • AM -> NM: Relies on NM to launch, monitor and stop allocated containers.
  • NM -> RM: Registers itself to RM for cluster membership(registration).
  • Containers -> NM:  for resource limits, status updates, and start/stop rules.

        While the concrete YARN architecture implements the four core components, our source-code analysis revealed a significant change in structure in relation to layering, modularity, and the communication structure between components.

        The RM and NM now sit in the server layer while the AM is bound to the client layer. And while the AM does have dependencies on both the NM and RM, the actual communication between these components is bound to the YARN-API. Both communication protocols used to send the RM and NM RPC’s are contained in the YARN-API.

YARN Concrete Architecture

Component Structure & Layers:

  • Server layer: ResourceManager, NodeManager and the real Container implementation.
  • Client layer: ApplicationMaster and Client. The interface to users.
  • Support layer: YARN Common, YARN-API (Shared data structures, protocol interfaces)

        

YARN concrete architecture also has 4 main components as our conceptual architecture: ResourceManager, ApplicationMaster, NodeManager and Container. However, we found that the developers applied many different design principles to make it more maintainable, flexible and scalable. This leads to a completely different architecture than our conceptual diagram. The container surprised us the most as we did not expect it to be inside of NodeManager instead of being a stand alone module. We found that there are 2 types of Container - logical representation and real worker executing tasks. The real Container is implemented inside NodeManager, and it’s used only by NodeManager. The logical Container is defined inside the YARN api and it’s used by all modules. This promotes separation of concerns as well as encapsulation.

        Communication between components are somewhat the same, but the implementation is much different as the developers made use of dependency inversion. All communication now implements protocol interfaces from YARN-api that would exchange request and response objects. These objects as well as record data structures inside them are all defined in YARN api. Only communication between NM and containers does not implement protocols but it still relies on data structures in YARN api.

YARN workflow

Workflow:

  1. Client submits a job to Resource Manager through ApplicationClientprotocol.  
  2. Resource Manager accepts the job and allocates a container for Application Master of the job application. It uses ContainerManagementProtocol to send ApplicationMasterContext to NodeManager
  3. Resource Manager asks the Node Manager of the node containing the allocated container to start it.
  4. Node Manager launches the container as well as Application Master on it. Node Manager will then send back a heartbeat to the Resource Manager, reporting a change in its containers’ status.
  5. Application Master registers itself to Resource Manager. After that it requests then retrieves containers’ information from the Resource Manager using ApplicationMasterProtocol.
  6. Application Master uses ContainerManagementProtocol to ask Node Manager to launch and assign tasks to those containers.
  7. Node Manager gets instructions from Application Master, resource from Resource Manager and launches the containers then assigns tasks to them.
  8. Anytime there’s a change in container status, NM sends a heart beat to RM using ContainerManagementProtocol.
  9. ResourceManager sends a response back to the client. When the app is finished, AM updates RM using ApplicationMasterProtocol and RM will tell NM to shut down the containers
  10. During execution, SharedCacheManager provides cached resources to reduce redundant data transfer. NM and yarn-client access it using clientSCMProtocol and NodeManagerSCMProtocol. TimelineService records runtime metrics, application states, and events and it is accessed using REST API

Lessons learned

        We reinforced our conceptual understanding of Hadoop architecture and also learned about practical implementation details: Common depends on HDFS for Windows - specific “unistd.h” as Windows are not POSIX compliant, thus not providing “unistd.h” for Common to compile its libraries. There are also test dependencies between layers. For YARN, we learned how modules are separated into different concerns (client, server, support) and the use of microservices architecture to promote maintainability, flexibility and scalability. We also learned how developers used dependency inversion with YARN api for all communications between modules. For the Container component, we discovered how developers applied separation of concern and encapsulation for logical representation and real working containers.
